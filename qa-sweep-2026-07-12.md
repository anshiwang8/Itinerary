## Method / environment
- **Primary sweep:** local `:3200` preview (`itinerary-verify`), full server-log visibility (`[schedule-resolve]`, `[swap-apply]`, `[reroute-apply]`, `[timezone]`). Port 3000 untouched.
- **Production parity:** scenarios 1, 2, 14, 15 re-run against `https://itinerary-six.vercel.app` — no runtime-log access there, so those compare rendered output only and avoid unsupported code claims.
- **In-app browser timezone:** `America/Toronto` (EDT, −04:00); real clock during the sweep ≈ 15:50 EDT on 2026-07-12.
- **Time simulation — important nuance discovered up front:** the dev time-sim picker (`simNow` in `app/page.tsx`) is only threaded into the itinerary **GET/reroute/swap** `?now=` calls (status/floor derivation) — see `page.tsx:453,477,543`. **Plan _creation_ (`resolveStartTimeChecked`/`buildSchedule`) always uses the browser's real `new Date()`** (`page.tsx:310,388,401`), so the picker cannot simulate a creation-time "now". For creation-time scenarios (2, 3, 14, 15) I therefore scoped-override the browser `Date` (verified: `new Date()`→fixed instant, `new Date(arg)` still parses) around the Plan click and reset it after. This is called out because it means the server-side `[schedule-resolve]` log still prints the real server time for those runs; the displayed schedule (client-side `buildSchedule`) reflects the simulated now.

### Scenario 1 — Baseline happy path

**Query / sequence:** City `Toronto`; prompt `ramen then a quiet bar in Ossington`. No special timing (real now ≈ 7:00 PM resolved / real clock ~3:50 PM EDT).

**Expected:** A two-stop Toronto plan — a ramen spot then a quiet bar in Ossington — with real venues, Toronto (EDT) times, a home "leave by" leg, `zone=America/Toronto`. Clarify skipped (category + aesthetic present).

**Actual:** Clarify correctly skipped. Log: `[schedule-resolve] time_window="unspecified" categories=["ramen","bar"] zone=America/Toronto resolved=2026-07-12T23:00:00.000Z` (7:00 PM EDT). But the plan rendered **one** stop: **BAR — 222 Bar, 7:23 PM – 8:33 PM** (4.0★), reached by a `505 Dundas` transit leg (leave by 7:00 PM → arrive 7:23 PM). The **ramen stop is absent from the strip**. The stored itinerary confirms it exists as `{category:"ramen", id:null, status:"skipped", start:null}`. Direct reproduction of the pool/select step: `ramen` pool size **0** (Places returned a single ramen result, "Crafty Ramen", dropped by rule `businessStatus = CLOSED_PERMANENTLY`); `bar` pool size 17; select returned `ramen → {id:null, reason:"no venues survived filtering"}`. The bar half is fully correct (real Ossington venue, Toronto time, home transit leg).

**Match:** No — one stop instead of two; the requested "ramen" silently vanished with no user-facing note.

**If No — code hypothesis:** Not a crash — a by-design partial-plan path. `app/page.tsx` runPipeline's empty-map net (≈ lines 348–358) only fails loud when **every** pool is empty (`poolEntries.every(arr => empty)`); a *partially* empty result (ramen 0, bar 17) passes the net and proceeds. `selectVenues` returns `id:null` for the empty ramen category ("no venues survived filtering"), which becomes a `status:"skipped"` stop in `createItinerary` (`store.ts`), and `ItineraryStrip` does not render skipped/null-id stops. Net effect: when one requested category has no open venues, the plan degrades to the surviving categories **with no banner or note** that a category was dropped. Root data cause here is Places coverage (only one "ramen" match near Ossington, permanently closed) — but the silent degradation is a pipeline/UX behavior, not a data artifact.

### Scenario 9 — Hard constraint, honestly unmet

**Query / sequence:** City `Toronto`; prompt `vegan steakhouse`. Clarify showed only "When?" (vibe correctly suppressed by the `vegan` constraint) → Skip.

**Expected:** The unmet-constraint refusal — "Couldn't find a … that's really vegan …" — matching the documented/`@mock` "vegan steakhouse fails honestly" behavior; never a hedged pick.

**Actual:** A **plan** rendered: **VEGAN STEAKHOUSE — Avelo Restaurant, 7:21 PM – 9:06 PM** (4.6★, editorial "…locally sourced vegan fare…"), reason "…a great choice for vegan options…". No hedge language. Parse of `vegan steakhouse` = `category_signals:["steakhouse","vegan"]`, `constraints:["vegan options"]`.

**Match:** No — a plan instead of the honest refusal (though notably it is *not* a hedged pick: Avelo is a genuinely vegan restaurant with real evidence).

**If No — code hypothesis:** The `@mock` "fails honestly" test uses a fixture that returns `id:null + unmetConstraint`; live behavior differs because of how real Groq decomposes the phrase. In `selectVenues`, the unmet-constraint/hedge guard fires only when the **constraint** (`vegan`) has no evidenced candidate or the pick hedges. Here the constraint is `"vegan options"`, which Avelo satisfies with strong evidence (its editorial summary states "vegan fare"), so the pick is valid and hedge-free. The contradictory facet — "steakhouse" — is carried as a **category_signal / search term**, not a hard constraint, so its non-satisfaction (Avelo is not a steakhouse) is never evaluated. Net: the system enforces "vegan" (met) and treats "steakhouse" as a searchable kind, so an impossible combination resolves to a real vegan venue rather than the documented refusal. This is a parse/constraint-modeling gap (what counts as a hard constraint), not a failure of the select hedge-guard.
