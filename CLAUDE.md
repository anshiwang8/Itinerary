# CLAUDE.md — Itinerary

Operational rules and invariants. Read this before making changes.

## What this is
Hyperlocal AI day-planner. Free-text prompt → one executable outing (real venues, real times, real transit), rendered on a map, that heals itself when transit breaks. The user supplies the CITY and starting address (Toronto/Ossington is the prefilled default and the pre-multi-city fallback, no longer the only target); each plan carries its own resolved IANA timezone. Next.js, Groq (Llama 3.3 70B), Google Places/Routes/Weather/Maps.

## Core architecture rule (governs everything)
The LLM does semantic work only: parsing prompts, selecting venues, writing reasons. Code does every verifiable fact: hours, prices, ratings, distances, travel times, scheduling. Never let the LLM compute distance/time or check "is it open" — that's a field comparison, not a judgment call. If a change asks the LLM to do something checkable, that's the bug.

## The pipeline (in order)
parse → weather fetch → Places search (one per category) → objective filter → LLM select → durations → travel (computeRoutes) → schedule → map. Each step is its own module and was built/tested in isolation.

## Non-negotiable invariants
- **Reroute floor_time:** floor_time = max(now, active stop's end). Stops at or before floor_time NEVER change. Locked stops never change. If any change violates this, STOP and flag it — do not paper over it.
- **Reroute blast radius + anchor:** a reroute replans ONLY stops strictly downstream of the disrupted leg (timed positions ≥ legIndex+1); stops upstream of the break keep their COMMITTED times even when the outing hasn't started. The replanned chain departs at max(floor, previous kept stop's committed end) — never re-derive the schedule from `now`; the clock only wins once it has overtaken the plan. `anchor_time` (that departure instant) is what the banner shows; `floor_time` stays the guarantee value. The `[reroute-apply]` log prints committed starts, floor, affected set, and post-reroute starts — check it first when a reroute moves the wrong stops.
- **Locked ratchet:** a stop locks when active or past, and never unlocks — survives completion and backwards time travel.
- **Keep-on-missing-data:** never drop a venue for missing data (no price, no hours, no rating = keep). Applies to every filter rule.
- **ID validation on select:** the LLM picks by venue ID only; invalid ID → one retry with correction → highest-rated fallback flagged. It can never invent a venue.

## Reuse, don't fork
The replan reuses the real pipeline via `searchPlaces.ts` and `selectVenues.ts` (callable cores; the routes are thin wrappers). Reroute imports these — never fork a parallel replan path, or fixes drift.

## Standing policies
- API keys server-side in `.env`, one per Google service, restricted in Cloud Console. Only the Maps JS key is browser-side (`NEXT_PUBLIC_`), protected by referrer restriction.
- Unstated times resolve via CATEGORY_START_DEFAULTS (dinner 19:00, brunch 10:30, etc.) before falling to next-full-hour. Past times roll forward to next day.
- One resolved `startInstant` is shared by scheduling, hours filtering, and the weather gate — they must never diverge.
- **Per-plan timezone (zone-aware everywhere):** every plan carries one resolved IANA zone (`itinerary.timeZone`, from `zoneFromLatLng` on the geocoded start; default `America/Toronto`). ALL scheduling math ("now"/"today"/which-hour), the hours filter's day-of-week + hour, and every display label render in THAT zone — never the server's wall clock, never the viewer's browser, never unconditionally Toronto. Zone math lives ONLY in `app/lib/zoneTime.ts` (luxon + tz-lookup, both offline); do not hand-roll `Intl.DateTimeFormat` parsing — that caused prior TZ/meridiem bugs. `toZonedISO(d, zone)` is byte-identical to the old Toronto formatter for the default zone, so Toronto plans are unchanged. When threading a new time computation, pass the plan's zone (`itinerary.timeZone ?? DEFAULT_ZONE`); `wallClockParts().weekday` is 0=Sun (NOT the day-of-month `day`). Note: `TZ=America/Toronto` on the server no longer drives scheduling (it's per-plan now) and `new Date()` is an absolute instant, so the server's zone is mostly irrelevant — with one real exception: the swap engine's default availability check (`usableByHours` in swap.ts) still reads `when.getDay()/getHours()`, i.e. the SERVER's wall clock, not the plan's zone. Keep the DEPLOY.md `TZ` check until that's fixed (logged in `code-audit-2026-07-18.md`).
- Transit legs get a +5 min delay margin; that buffer is separate from stop-duration buffers. A leg relabels as walk (no margin) when it's short (<400m), or when walking beats transit+margin AND the walk is ≤30 min (`MAX_WALK_LABEL_MIN`), or when walking is at least twice as fast as transit (broken-transit escape hatch, any length) — a 75-minute walk must never be presented over a similar transit ride.
- **Location (multi-city):** the CITY and STARTING ADDRESS are plain text inputs on the empty state — geocoded via `POST /api/geocode` (reuses the Places key; no separate Geocoding API). The city rides on `parsed.city` (app-injected, never LLM-inferred) so swap/reroute re-searches inherit it; the geocoded start is stored as `itinerary.home` and everything home-related reads `itinerary.home ?? HOME` (the Chestnut default exists ONLY for pre-multi-city itineraries and tests). Weather takes `?lat=&lng=`. The parse's `location` field is a neighbourhood WITHIN the city, never a city. Never silently fall back to Ossington/Toronto for a user-entered city. Real location services / geolocation remain deliberately unbuilt — plain queries only.
- **Persistence seam:** routes access the itinerary store ONLY via `loadItinerary`/`saveItinerary` (store.ts). KV env vars set → Upstash Redis REST is the source of truth (always read Redis, always write back after mutations — `withStatuses`' lock ratchet mutates even on GETs and refusals); unset → the in-memory Map (dev/e2e). Engines never touch the store. Serverless without KV fails loud. Deployment (env table, Maps-key referrer restriction, TZ=America/Toronto requirement) lives in `DEPLOY.md`.

## Fail-loud surface (operational notes)
`app/lib/planGuards.ts` is THE surface for bad input — every degenerate/impossible/contradictory prompt must land there (or in `resolveStartTimeChecked`'s band messages) with a reason + suggested fix; an empty map or a borrowed error from another branch is the bug. Guard order in `runPipeline`: degenerate prompt (pre-Groq) → unparseable parse → contradiction → [clarify questions] → time band → all-pools-empty net → unmet constraint → PARTIAL-failure recovery (the interactive panel below). A hard refusal string is the right answer only when there's no real choice to offer; once there IS one (an inferred-time guess, a weather block, a partly-empty plan), it belongs in the recovery panel instead — see the next section. Constraints are enforced in `selectVenues`: no evidenced candidate → `id:null + unmet_constraint` (valid, no fallback), and a code-side hedge guard converts "worth confirming / may accommodate" picks into the same failure — never suggest a venue while telling the user to verify a constraint. `priceLevel` + `description` ride ON the stop (Selection → SelectionLike → ItineraryStop, set by swap's buildStop/finalize too); the strip must read the stop's own fields before any pools lookup, which goes stale after swaps.

## Shipped — interactive recovery panel (operational notes)
ONE component in `app/page.tsx` (`recovery` state, a discriminated union on `mode`), three triggers. Never build a parallel panel for a fourth case — add a mode.
- **`mode: "empty"`** — partial failure: some categories picked, ≥1 pool came back empty. `partialEmptyCategories` detects it (all-empty stays on `noVenuesReason`, never routes here). Each row shows `emptyCategoryReason` (the honest dominant reason from the objective drop log) + a widen offer (re-search city-wide via `categoriesOverride`, suppressed per-row by `noWiden`) + a free-text replace + "Plan without it" (shown only when some other stop actually resolved).
- **`mode: "time-gate"`** — the user typed NO time and our own inferred slot fell outside a known category band. `resolveStartTimeChecked` marks ONLY this path `overridable: true` (+ `category`); "still want it" re-enters `continuePipeline` with `overrideTimeGate`, which skips the band check for that run. **This distinction is load-bearing: an explicitly stated impossible time (`implausibleExplicitReason`, "brunch at 3am") is NEVER overridable.** "Something else" re-opens the kind question with `time_window` stamped to an explicit `"now"` — otherwise the new category's default (bar → 20:00) has already passed and rolls the plan to tomorrow.
- **`mode: "weather-gate"`** — a category was emptied by the weather gate specifically, while others survived. "Still want it" re-searches that category with `weather: null`, which skips ONLY the weather gate via keep-on-missing — hours/rating/price/business status all still run, and no server change was needed. Resolved categories must be cleared from `weatherBlocks` (a planned stop is not "skipped"). Still-empty falls into `mode: "empty"`, never a third dead end. "Something else" scopes to the SLOT (replace rows with `noWiden`), not the whole-plan kind picker — other stops survived and discarding them would be wrong. The ALL-weather-blocked path keeps `weatherBlockedReason` untouched.
- Shared plumbing: `searchSlot` (one category search+select, with `dropLocation`/`ignoreWeather` switches) and `mergeSlot` — widen, replace, and both overrides go through the same fetch path. `orderByRequest` restores prompt order afterwards, because `selectVenues` appends empty categories LAST and recovery resolves them in that appended position.

## Shipped — swap engine (operational notes)
Per-stop swap (`app/api/itinerary/swap.ts`, `POST /api/itinerary/[id]/swap`): user taps an UPCOMING stop, types a complaint, the engine acts on intent — VENUE (replace, hold slot), TIME (move the slot), DURATION (change how long), or CONSTRAINT (re-search). Distinct from reroute (external disruption → downstream replan). `floorTime()` shared from `store.ts` by both engines.
- **Time / duration swaps**: `parseTimeExpr` (relative "an hour earlier" = -60, absolute "after 8" = 20:00, meridiem anywhere "6pm" = 18:00, bare "3:30" = 15:30 — assume PM only for meridiem-less times; a number owned by a duration unit, "make it 2 hours", is never a clock time) and `parseDurationExpr` (absolute "stay 2 hours" = 120, relative "stay longer" = +30) are the deterministic floors under the Groq interpret — arithmetic never depends on the model. Merge rules: a locally-parsed ABSOLUTE time always beats the model's; an EXACT local relative beats the model's; only a VAGUE local relative ("a bit earlier") accepts model refinement, and only same-sign ≤3h — garbage deltas must never drift a stop across the day. A phrase with BOTH halves ("start at 6pm for 2 hours") routes to time and applies the duration alongside via the shared `resolveNewTotal` bounds — neither half is ever dropped. A time-swap moves the slot, never resizes it, and a VENUE swap holds the whole slot, length included: the stop's customized `durationMinutes.total` is the source of truth everywhere a swap keeps the same category. It resets to the (new) category default only when the stop changes KIND — a category-changing research swap, or an engine-adapted replacement venue. The `[swap-apply]` log in `swapStop` prints intent + before/after start/total — check it first when a swap changes the wrong field. Both call the reusable **try → adapt → notify** ladder `resettleTail` (shift/reflow the tail, keep a downstream venue if still open at its new arrival, else re-search an equivalent, else fail loud). Plan-then-commit: a failed resettle leaves the itinerary untouched. Duration guards: floor at the category's realistic minimum, cap 6h. `buildStop` takes a `durationOverride`; `resettleTail`/`commitTail` carry each stop's `totalMinutes` so a custom duration survives a later reflow. Keep `resettleTail` generic — it's the shared spine of time + duration (and future) swaps.
- **Availability seam**: `SwapDeps.isUsableAt(place, when, category)` is the ONE "is this venue usable then" check (default = objective hours, keep-on-missing). A real availability/reservation API replaces that function body only.
- **UI**: the swap prompt is inline under the selected strip card (`ItineraryStrip` `SwapInline`), upcoming-only; reuses the reroute reflow (chartreuse "changed" keyed by venue id). The map's projection probe defers its `setState` to rAF (never call setState inside a Google `OverlayView.draw()`).

## UI invariants (don't regress)
- Acid green (`--live`) is reserved. The FULL chartreuse marker/border = the active "now" stop only. Swap-/reroute-changed elements get chartreuse *accents* (changed-ring marker, settling time pill, redrawn route) but an upcoming changed stop keeps its ink marker — chartreuse never means "changed" and "now" at the same visual weight. Never decoration.
- Swap vs reroute: swap is user-initiated + surgical (one upcoming stop, hold the slot); reroute is disruption-driven + replans the tail. Both share `floorTime()` and reuse the same pipeline cores — never fork a third replan path.
- Map uses inline `styles` JSON on a classic `Map` (no `mapId`) so paper styling works; markers/cards are `OverlayView` HTML, NOT AdvancedMarkerElement (which needs a Cloud map id).
- fitBounds keys off venue coordinates (`fitKey`), never status ticks — status changes must not yank the map view.
- Expanded cards clamp to the viewport (horizontal) and flip below the pin near the top edge.

## Open gaps (deferred, not blockers)
Each item below was re-verified against current code on 2026-07-18. These are DEFERRED SCOPE, not defects — actual defects live in `code-audit-2026-07-18.md`.
- **Reservation availability** — needs OpenTable/Resy. The seam is ready: `SwapDeps.isUsableAt` is the one "can we use this venue then" function; a real API replaces that body only.
- **GTFS-realtime** — the reroute engine is real, the trigger is the dev button. No rideshare fallback (`travel.ts` computes TRANSIT and WALK only).
- **Movie runtimes** — `durations.ts` still carries the 120+30 placeholder and its TODO; real runtimes need TMDB/showtimes.
- **One city per plan** — city+address are plain query inputs; a prompt spanning two cities still plans in the entered city. Real location services/geolocation are future work. (Per-city display timezones ARE resolved, as of Phase 5.)
- **Dev `?now=` picker reads the VIEWER's browser zone** — simulating time on a non-local-zone plan is offset. Dev control only: status derivation is absolute-instant correct, and mock e2e is unaffected.
- **Stop reordering** — not implemented.
- **Pick reasons don't see resolved times** — `selectVenues` receives the parse, not the schedule, so a reason can't refer to a stop's final time.
- **Reroute AND swap skip the weather gate** — both pass `weather: null` to `filterPools` (reroute.ts has the explicit TODO; swap's `venueSwap`/`findReplacement` do the same). Only the initial plan consults the forecast. Corrected wording: this was previously listed as reroute-only.
- **Chips can tuck under an expanded card** when two venues are <~300m apart (the active card wins the z-order).
- **A DECLINED weather block still has no pin** — it renders as a text note, since a skipped category has no venue or coordinates. Narrowed: this used to read as "weather-blocked categories have no recourse"; that part is **RESOLVED** — a partial weather block now opens the weather-gate panel (see the recovery-panel section), and an accepted override plans a real stop with a real pin. The no-pin note applies only to blocks the user declines or that stay empty.
- **Mock e2e for a second city** — `mockGeocode` is city-agnostic, so a mock scenario would prove plumbing only. Multi-city is live-verified instead; still flagged as a follow-up.
