# Devlog — Itinerary

## Feature: home origin (leg 0 from Chestnut Residence)
Goal: itineraries start from the user's home — hardcoded to UofT Chestnut Residence (89 Chestnut St, 43.6547, -79.3862) — instead of materializing at the first venue.
What should be done: prepend a home → stop 1 travel leg using the existing computeRoutes logic (transit-first, walk fallback, +5 min margin, short-leg relabel all unchanged); the resolved start becomes the leave-home time and stop 1 starts after the leg; home is an origin waypoint, not a stop (no duration, no card, excluded from stop count/statuses/completion); render a distinct house marker and a "from Home · Chestnut Residence — mode · X min" line; reroute must never touch leg 0, without special-casing the engine; venue search stays anchored on Ossington.
What was done (technical): new `app/api/schedule/home.ts` with the HOME constant, HOME_LEG_INDEX sentinel (-1), and pure `splitHomeLeg` that splits `getTravelLegs([HOME, ...venues])` output into the home leg and re-indexed inter-stop legs. `buildSchedule` gained an optional `homeLeg` param: startISO = leave-home time, first timed stop starts `homeLeg.totalMinutes` later (mirrors the reroute engine's floor + inbound anchoring). The itinerary store persists `homeLeg` as a separate field — deliberately outside `stops` and `legs`, so status derivation, completion, the disruption dropdown, and the reroute engine's leg-index math structurally never see it (no special-case needed; a reroute-after-departure test pins it byte-identical). PlacesTest sends home as point 0 of the same travel request (single-stop plans now get a home leg too) and renders the leg-0 line; ItineraryMap gained a `home` prop with a dark house-glyph pin and the same solid-walk/dashed-transit leg styling. Live run confirmed a genuine named transit leg 0: 505 Dundas, 11 stops, boarding at Dundas St W at Chestnut St, ~28 min incl. margin. 75 tests green across 8 suites (4 new home tests, +2 schedule, +1 itinerary, +1 reroute).

## Fix: rolled-forward schedules show the date
Goal: tomorrow's 10:30 must not read identically to today's — rolled-forward (or any non-today) stops need a date on the "be here" line.
What should be done: per-stop, when a resolved start is not today, prefix the time with a date ("tomorrow" for +1 day, else a short "Sat Jul 6"); same-day stays time-only. Decide per resolved instant, not per itinerary, so a schedule crossing midnight labels only the post-midnight stops. Banner floor time and home-leg leave time follow the same rule. Frontend + formatting only — no scheduling logic changes.
What was done (technical): new pure module `app/lib/timeLabels.ts` — `datePrefix` (calendar-day offset vs a reference now → "" / "tomorrow, " / "Sat Jul 6, "), `formatStopTime` (single labeled instant), `formatStopRange` (the "be here" range, prefix keyed off the START only so a stop straddling midnight stays today-labeled). Wired into the "be here" line, the reroute banner's floor time, and a home-leg "leave by" time (= first timed stop's start − home leg travel). Live-verified: "brunch tomorrow" → "be here tomorrow, 10:22 AM – 12:07 PM" / "leave by tomorrow, 10:00 AM"; "dinner tonight" → "be here 8:29 PM – 10:14 PM" / "leave by 8:00 PM" (time only). 82 tests green across 9 suites (7 new timeLabels tests, incl. the crossing-midnight case).

## Feature: duration swaps (third swap type)
Goal: handle "how long" ("stay 2 hours", "stay longer", "shorter", "just an hour") as a third swap intent, reusing the existing try→adapt→notify ladder.
What should be done: add intent "duration" with a deterministic `parseDurationExpr` floor (absolute "stay 2 hours" = 120, relative "stay longer" = +30, "an extra hour" = +60, "shorter" = -30); change the stop's duration (end = start + new duration, start stays), then reuse `resettleTail` to reflow/adapt downstream; sane bounds (floor at the category's realistic minimum so "10 minute dinner" fails, cap at 6h so "stay 20 hours" fails); product-voice banner; reuse `isUsableAt` (a longer stay just means later downstream arrivals, already covered). Do NOT write new tail-rechaining or availability logic.
What was done (technical): `SwapInterpretation.duration` = `{ mode: "relative"|"absolute", deltaMinutes?, targetMinutes? }`, with `parseDurationExpr` (exported, unit-tested) as the floor under the Groq call — duration wins over time, and a bare time direction with no duration word ("an hour earlier") is explicitly NOT read as a duration. New `durationChange` computes the new total from the stop's current duration, guards it (categoryMin = max(15, 40% of the category default) so "10 minute dinner" fails; cap 360 so "stay 20 hours" fails), then calls the SAME `resettleTail` used by time-swaps. `buildStop` gained a `durationOverride` (custom total, category buffer preserved); `resettleTail`/`commitTail` now carry each stop's `totalMinutes` so a kept stop's duration survives a later reflow (adapted stops take the category default). Availability seam and the ladder are untouched — no duplication. Banner in product voice: "Extended dinner to 2 hours — everything after shifted later" (adds "(moved a later stop to a spot that's still open)" when it adapts). Tests: 8 new cases (parseDurationExpr; "stay 2 hours" extend+reflow; "stay longer" +30; "shorter" -30; extension that closes a later venue adapts; extension nothing absorbs fails loud; "stay 20 hours" + "10 minute dinner" both fail loud; locked downstream never moves) — 22 swap cases, 104 green across 10 suites, tsc clean. Live-verified: "stay 2 hours" extended dinner 8:29→10:29 PM and reflowed the bar (strike→settle); "stay longer" took it to 2h 30m and reflowed again.

## Fix: time-swap intent + inline swap prompt
Goal: "an hour earlier" returned "couldn't tell what time you meant" — relative time expressions didn't parse. Fix the time path (venue swaps already worked), and move the swap prompt off the detached bottom bar into the selected strip card.
What should be done: interpret must handle relative ("an hour earlier" → -60, "a bit later" → +30, vague defaults to a 30-min step) AND absolute ("after 8" → 20:00); apply the shift, then a general try → adapt → notify ladder (shift + reflow via shared chaining, floor_time guard; if a later stop closes at the new timing, re-search an equivalent; only if nothing works, fail with the specific reason). Make the ladder a reusable helper (duration-swaps reuse it next). Move the swap prompt inline under the selected card. Isolate the "is this venue usable at this time" check behind one seam for a future availability API.
What was done (technical): `SwapInterpretation.time` is now `{ mode: "relative"|"absolute", deltaMinutes?, targetTime? }`; a deterministic `parseTimeExpr` (exported, unit-tested) parses relative/absolute expressions and is the floor under the Groq call — a time phrase stays a time request even if the model misclassifies it. The engine's `timeChange` computes the new start, guards (plausible hour → floor → no overlap with the previous stop), resolves the anchor venue (keep if usable at the new time, else adapt, else notify), then calls the new reusable `resettleTail(itinerary, anchorIndex, …, anchorEnd, anchorLoc, …, used)` — the try→adapt→notify ladder that re-chains each downstream stop, keeping its venue when usable and re-searching an equivalent (`findReplacement`) when it closes, failing loud with the real reason if nothing adapts; the whole plan is computed before any mutation, so a failure leaves the itinerary untouched. Availability is isolated behind `SwapDeps.isUsableAt(place, when, category)` (default `usableByHours` via `isOpenAt`, keep-on-missing) — a real availability API replaces that one function body. `bandForCategories` / `isPlausibleAt` were exported from `schedule.ts` for the plausible-hour guard. Frontend: the swap prompt moved into `ItineraryStrip`'s selected card (`SwapInline`), the detached bottom `.swapbar` is gone; the banner no longer double-prefixes time results ("Moved bar to 10:48 PM", not "Swapped bar — Moved bar…"). Also fixed a map crash: the projection probe called `setState` inside `draw()`; it's now deferred to `requestAnimationFrame`. Tests: 14 swap cases (parseTimeExpr relative/absolute/non-time; -60 reflow; +30 default; absolute "after 8"; adapt-on-close; fail-when-nothing-adapts; locked downstream never moves; "dinner at 4am" fails loud) — 96 green across 10 suites, tsc clean. Live-verified end to end: "an hour earlier" moved dinner 8:29 → 7:29 PM and reflowed the bar; "a bit later" moved the bar 10:18 → 10:48 PM. Duration/extend deliberately untouched — the ladder is left generic for it to reuse.

## Feature: per-stop swap
Goal: let the user tap ONE upcoming stop, type a short complaint ("somewhere cheaper", "with a patio"), and replace just that stop with a closer match — surgical, distinct from reroute (which replans the whole downstream tail after an external disruption).
What should be done: an engine that guards to upcoming-only (reuse floor_time), interprets the complaint via a small Groq call into an updated constraint set + a re-filter (narrow the same pool) vs re-search (new query for features/cuisine the pool lacks) classification, excludes the rejected venue, holds the stop's time slot, recomputes its duration + outbound leg, and shifts downstream only when the new pick genuinely can't fit — never touching locked/past. Reuse searchPlaces/selectVenues/filter/schedule cores; return a diff; wire a POST route; add an inline swap prompt that reuses the reroute reflow visuals; product-voice copy; honest refusal when nothing better is found.
What was done (technical): shared floor_time by extracting `floorTime(itinerary, now)` into `store.ts` (reroute now calls it too — behaviour identical, its 7 tests unchanged) and exported `toTorontoISO` from `schedule.ts`. New `app/api/itinerary/swap.ts`: injectable `SwapDeps` (interpret / searchPools / selectVenues / getSingleLeg); a Groq `interpretRefinement` returning `{ path, category, aesthetic, budget, constraints }`; refilter searches the ORIGINAL query then judges with the new constraints, research searches with the new query too; rejected + already-used venue ids excluded before select; the new venue holds the old `start_time`, duration recomputed via `getDuration`, inbound (map geometry) and outbound legs re-fetched; fit check shifts the downstream tail by the delta via `toTorontoISO` only when it overflows (locked/past sit before the swapped stop and are never touched); returns `{ swapped, stopIndex, path, before, after, reason, downstreamShifted }`; empty candidates → honest `{ swapped:false, reason:"Couldn't find a better …" }`. `POST /api/itinerary/[id]/swap` wraps it. Frontend: an ink-navy swap bar revealed only when an upcoming stop is selected ("Not quite right? Change <venue>"), `doSwap` reusing the reroute reflow — "changed" is now keyed by venue id (a swap can change a stop's category), the swapped stop gets a chartreuse changed-ring marker (NOT a "now" pin, since it's upcoming) and settles into place, downstream shifts strike-through their old times; banner "Swapped dinner — <reason>". 6 swap unit tests (held slot, downstream shift + locked untouched, rejected never reappears, refilter vs research classified, active/past rejected, nothing-better refusal). Live-verified: "somewhere cheaper" on the dinner swapped Union → Salt Wine Bar Ltd, held the 8:29–10:14 PM slot, chartreuse changed accent; moving the dev clock into the stop made it the active "now" (full chartreuse marker) and hid the swap bar. 88 tests green across 10 suites, tsc clean. Pipeline cores untouched.
